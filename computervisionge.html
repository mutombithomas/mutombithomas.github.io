1. Background and Commercial Incentive

Large area displays in cockpits are becoming increasingly common in modern designs for commercial aircraft to increase ease of operations and to make efficient use of limited cockpit space[8][9][10]. Touchscreen sensors alone cannot identify when multiple users are interacting with the touchscreen or which touch coordinates belong to each user[12]. For this reason, there is a growing need to use computer vision techniques to automate user recognition to avoid inadvertent commands that are not acceptable in the safety critical Aerospace domain.


2. Object Tracking

2.1 Investigation Overview

An investigation into object tracking algorithms as part of python’s OpenCV library was performed in order to gauge the performance of standard object algorithms. Using the vertical camera position, a candidate set of tracking algorithms were investigated and are briefly described in table 1.

OpenCV Algorithm Description
BOOSTING At runtime, the classifier is trained via two bounding boxes: a positive instance as well as a negative instance of the object to be tracked. During runtime, classification is run on the current bounding box pixels as well as a neighbourhood of pixels[3].
MIL (Multiple Instance Learning Similar to the BOOSTING algorithm with an emphasis on positive instances instead of negative ones .
TLD (Tracking-Learning-Detection) Instead of learning features from the first frame, the algorithm updates changing features during tracking progression [5].
KCF (Kernelized Correlation Filter) The KCF learns a model of the object from the first frame and subsequently attempts to locate the same model in future frames[4].
CSRT (Channel Spatial Reliability Tracker) Uses a Discriminative Correlation Filter with Channel and Spatial Reliability [7].
MEDIANFLOW Tracks a given object using the backward and forward frames with respect to the current one [6]

2.2 Evalution Methodology

Evaluation of each object tracking algorithm was performed by individually assessing the performance on 3 separate videos, each of which simulated realistic user interactions with the monitor that would be seen in the cockpit environment.

Figure 1. Demonstration CRST algorithm successfully tracking single touch after instantiating the classifier.

Evaluation of the tracking performance was based on the speed of the algorithm, as well as the precision and recall classification metrics defined below.

Precision=TPTP+FP Recall=TPTP+FN

Precision indicates the proportion of correct predictions out of all positive predictions, whereas recall provides a measure of how likely the algorithm is to make the correct prediction. An ideal algorithm would maximise these metrics to 1. Each prediction was classified in relation to a hand-labelled ground truth (GT) for each frame where a bounding box of constant dimensions defined a true positive. To gauge the similarity of a prediction to the GT, the Intersection Over Union (IOU) metric was used as shown below, where P is the predicting bounding box of the same dimensions as the GT.

IOU=GT∩PGT∩P

Depending on the threshold, T, set for the algorithm, IOU≥T determines a true positive and value below is classified as a false positive. If the tracker fails to give any prediction, a false negative is flagged. For all videos in the data set, thresholds T∈[0,1] were assessed. The area under the curve (AUC) was computed for each metric which is referenced in the results section.

2.3 Results & Discussion

Evaluation of the algorithms given in table 1 was completed for all hand crossing scenarios. The subplots shown in figure 2 display the precision of the tracking algorithms over the full range of IoU thresholds. The degradation of all trackers as the threshold was increased was the result of the stricter requirement to capture a greater proportion of the GT to assign a true positive classification. With regard to relative performances, the CSRT performed outperformed the other tracking algorithms with the highest average precision AUC score and for all 3 hand crossing scenarios at 0.58.

Figure 2. Comparison of precision of object tracking algorithms for all 3 hand motions: single touch (top left); non-crossing touch (top right); crossing touch (bottom).

A perfect recall score for trackers such as CSRT and MEDIANFLOW was indicative of their ability to avoid prediction failure, irrespective of whether the prediction was correct or not. This can be inferred from the recall definition where the metric is strongly influenced by the false negative count. Therefore, the all trackers that showed a non-unity score, contained instances where no predicting bounding box was projected. TLD consistently failed in this regard, as well as in the speed comparison.

Figure 3. Comparison of recall of object tracking algorithms for all 3 hand motions: single touch (top left); non-crossing hands (top right); crossing touch (bottom).

The rate at which algorithms were able to process each frame while tracking is shown in figure X by the FPS. Although CSRT was able to better predict the GT than all other algorithms, it could not match the others for speed. KCF and MEDIANFLOW were showed their ability to efficiently process each frame from their high FPS rate but both with severe fluctuations. The reason for this is could not be specifically identified but could be due to the computer hardware used for the investigation.

2.4 Object Tracking Conclusion

In general, the investigation into the use of OpenCV’s object tracking algorithms concluded that even the most accurate algorithm, was too erratic and slow for use in any system that demands reliability. Furthermore, the inability of any algorithm to deal with occlusion was a key factor in deciding not to pursue this approach any further.

3. Fingertip Detection & Occulsion

In addition to the assessment of OpenCV's tracking algorithms, a separate hand tracking methodology was developed and implemented using OpenCV's library of computer vision functions. These specifically focused on hand identification through pixel manipulation, as well as fingertip detection. Both methdologies are described below.

3.1 Hand Identification

After the infrared video feed was recieved by a python script, a perspective transform was applied to each frame to focus only on the touch screen display. A Gaussian blur was then applied to each frame. This is where each pixel is the sum of all the pixels within a specified radius, which is then multiplied by a gaussian kernel.

This causes small groups of white pixels to decrease in value while large groups remain relatively unaffected, resulting in noise and small objects being removed from the image when a threshold is applied. The resulting image has a black background with only hands and arms remaining.

To differentiate between objects that enter the frame, OpenCV's function findContours was used. This uses edge detection to return an array containing an outline of each individual object found within the image. However, in order to match objects between frames, Zernike moments were used, which outputs an array of values that describe the properties of the image, including its size, shape and colour distribution. This was used to compare the similarity of two objects. If the moments match between two objects in consecutive frames, they were identified as being the same object and assigned a colour, as shown in the figure. If multiple objects from the previous frame were matched to a single object in the current frame, the occlusion case was triggered, which is described in section 3.3.

Athough each object in the frame was able to be matched to the same object previous frames, OpenCV's SIFT feature detection was used to track the movement of the hand. Since each individual hand was identified, fingertip detection was implemented.

3.2 Fingertip Detection

Fingertip detection and tracking was implemented using contour detection and convex hull techniques. Using the output of findContours, the convexHull function was used to generate a convex hull. This can be described as the smallest polygon that will contain the contour detected. The convexityDefects function was then used to approximate the gaps between the contour and convex hull using straight lines. An example of this technique can be seen in the figure to the right.

To filter between fingertips and other identified defects, the inner angles between the convexity defect lines was calculated using the the following logic: if the inner angle was found to be ≤ 90 degrees the defect was treated as a fingertip and marked. If it was found to be larger the defect, it was ignored. This methodology allows for all fingertips to be tracked simultaneously.

3.3 Dealing with Occlusion

To deal with occlusion, a Gaussian mixture method model (GMM) was created for each hand found in the video feed. Between each frame, the hand was tracked and the hand in the current frame was translated to the position of the model. The current hand was then applied to the model and over time, as the hand moves into the frame, the model of the full hand is created. Figure 5 shown the progression of the GMM as the users hand moves into the frame; every 5th frame is shown as the movement between individual frames is minimal.

Figure 5. Gaussian Mixture Model progression.

By creating a model of the hand and tracking it between frames, when occlusion occurs, the model can be overlaid on the portion of the users hand/arm that’s visible. This then shows where the hidden portion would be. An example of this is shown in figure 6.

Figure 6. Occulsion example: the left image shows the output from the infrared camera where the left hand is obscured by the right hand. The right image shows the model of each hand that has been overlaid and therefore the position of the left hands fingertip can be identified.
